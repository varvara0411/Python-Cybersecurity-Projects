### **Детектив по логам: поиск веб-атак в логах Apache/Nginx**

##### *Описание задачи*

Веб-серверы, такие как Apache и Nginx, являются основной точкой входа для многих веб-приложений и, к сожалению, часто становятся мишенью для различных атак. Злоумышленники пытаются использовать уязвимости в коде приложений или конфигурации сервера для получения несанкционированного доступа, кражи данных или нарушения работы сервиса.

Логи доступа веб-сервера (обычно access.log) фиксируют каждый запрос, поступающий на сервер. Анализ этих логов позволяет выявить подозрительную активность, включая попытки распространенных веб-атак, таких как:
* SQL-инъекции (SQLi): Внедрение вредоносного SQL-кода в запросы для манипулирования базой данных.
* Межсайтовый скриптинг (XSS): Внедрение вредоносных скриптов (обычно JavaScript) в веб-страницы, которые затем выполняются в браузерах других пользователей.

**Логи доступа веб-сервера (Common Log Format)**

Стандартный формат логов доступа веб-серверов (Common Log Format, CLF) включает следующую информацию для каждого запроса:

IP_адрес - Идентификатор_пользователя [Время_запроса] "Метод /Путь_и_параметры Версия_HTTP" Статус_ответа Размер_ответа
                  
Пример:

192.168.1.100 - - [21/May/2023:12:34:56 +0000] "GET /index.php?id=1 HTTP/1.1" 200 1234
                 
Для анализа на предмет веб-атак нас в первую очередь интересует часть запроса: `"Метод /Путь_и_параметры Версия_HTTP"`, особенно `/Путь_и_параметры`.

**SQL-инъекции (SQLi)**

Атаки SQLi эксплуатируют уязвимости в обработке пользовательского ввода, который используется для формирования SQL-запросов к базе данных. Злоумышленник пытается внедрить специальные символы и команды SQL для обхода проверок или выполнения произвольных запросов.

Характерные признаки в URL:

* Одиночные кавычки (`'` или URL-кодированные `%27`)
* SQL-комментарии (`--`, `#`, `/* ... */`)
* Ключевые слова SQL (`UNION`, `SELECT`, `INSERT`, `UPDATE`, `DELETE`)
* Логические операторы (`OR`, `AND`), часто в конструкциях типа `OR 1=1`

Пример подозрительного запроса: `GET /products.php?category=1' OR '1'='1`

**Межсайтовый скриптинг (XSS)**

Атаки XSS заключаются во внедрении вредоносного кода (обычно JavaScript) на веб-страницу. Этот код выполняется в браузере пользователя, который посещает зараженную страницу, и может использоваться для кражи сессий, перенаправления на фишинговые сайты или выполнения других вредоносных действий.

Характерные признаки в URL:

* Теги HTML (`<script>`, `<img>`, `<iframe>`, часто URL-кодированные как `%3Cscript%3E`)
* Атрибуты событий JavaScript (`onerror`, `onload`)
* Функции JavaScript (`alert()`, `document.cookie`)

Пример подозрительного запроса: `GET /search.php?query=<script>alert('XSS');</script>`

**URL-декодирование**

Злоумышленники часто используют URL-кодирование для маскировки вредоносных символов. Например, `'` кодируется как `%27`, `<` как `%3C`. Перед проверкой на наличие паттернов атак необходимо декодировать URL-параметры. В Python для этого используется функция `urllib.parse.unquote()`.

**Регулярные выражения для обнаружения атак**

Для поиска признаков атак в строках запроса эффективно использовать регулярные выражения. Необходимо составить шаблоны, которые будут выявлять характерные для SQLi и XSS конструкции.

Важно: Создание надежных паттернов для обнаружения всех видов атак — сложная задача. В этой задаче мы будем использовать упрощенные паттерны для демонстрации основного принципа.

##### *Цель*

Написать скрипт на Python, который анализирует лог-файл доступа веб-сервера (в формате Common Log Format) и выявляет строки, содержащие признаки потенциальных SQL-инъекций или XSS-атак в URL запроса.

Программа должна:

1. Считать данные лога из стандартного ввода
2. Распарсить каждую строку лога, извлекая часть запроса (URL и параметры)
3. Декодировать URL-кодированные символы в запросе (например, `%27` в `'`, `%3C` в `<`)
4. Проверить декодированный запрос на наличие характерных паттернов SQLi и XSS с помощью регулярных выражений
5. Если в запросе найден хотя бы один подозрительный паттерн, вывести полную исходную строку лога
6. Вывести отсортированный список подозрительных строк лога, каждая на новой строке
